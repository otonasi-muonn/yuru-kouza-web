## `index.md`（トップ）

**狙い**: 参加者が最初に開いた瞬間、「何を今日できるようになるか」「このサイトの使い方」が腹落ちする。加えて、**このサイト自体が4/29で学ぶ技術の完成見本**になっていることを伝える。

**構成**
1. 歓迎メッセージ（1〜2文、堅苦しくしない）
2. 今日のゴール: 「自分のGitHubプロフィール画面に、自己紹介READMEが表示された状態」
3. 4/26と4/29の関係図（4/26は**土台作り**、4/29は**制作**）
4. **ドッグフーディングの明示**
   - 「このサイトも GitHub Pages で公開されています」
   - 「4/29 の Web 班で作るものは、このサイトと同じ仕組みで動きます」
   - 動機付け: 今日学ぶ「リポジトリ・Commit・Push」が4/29の公開に直結する
5. サイトの使い方
   - 進捗チェックの説明
   - OS切替の説明
   - **進捗リセット**の説明（万一おかしくなったらリセットボタンから初期化できる）
   - 「先に進みすぎず、メンターと歩調を合わせる」ガイド
6. タイムスケジュール（仕様書から転記）
7. 「スタート → 共通モジュール」ボタン

## 共通モジュール `common/`

### `01-github-account.md` GitHubアカウント作成

- 既に持っている人は**スキップ可**と冒頭で明示
- サインアップ画面 → メール認証 → ユーザー名の決め方
  - **重要**: ユーザー名は将来も使うので「適当な連番」や「一時的な名前」にしない
- よくあるつまずき: メール未達、強パスワード要件

### `02-vscode-extensions.md` VSCode推奨拡張

- 推奨拡張（最小限）
  - Japanese Language Pack
  - Markdown All in One
  - GitLens（任意）
- 拡張機能タブから検索 → Install
- 再起動が必要な場合がある
- よくあるつまずき: 検索結果に類似拡張が複数ある → 発行元を確認

### `03-github-oauth.md` GitHub OAuth連携

- VSCode左下の「アカウント」アイコン → 「GitHubでサインイン」
- ブラウザが開く → 承認 → VSCodeに戻る
- **SSHは使わない**と明記（挫折防止）
- よくあるつまずき: ブラウザが戻ってこない → 一度VSCode再起動 or 手動で `Ctrl/Cmd + Shift + P` から `Sign in to GitHub`

## Web班モジュール `web/`

### `01-create-repository.md` リポジトリ作成

**最重要**: 「ユーザー名と同名の**特殊リポジトリ**」の説明をスクショ強く、枠・矢印で強調。

- GitHub画面右上「+」→「New repository」
- Repository name に**自分のユーザー名と完全一致**を入力
  - 「特殊リポジトリ」と呼ぶ機能があり、プロフィール画面に README が表示される
  - 一致していないと表示されないことを強調
- **Public** 必須
- **Add a README file** にチェック必須
- Create repository

**よくあるつまずき**
- ユーザー名の大文字小文字を間違える → GitHubは大小区別しないが、視認性のため同一で
- Private にしてしまう → 表示されない、作り直す
- README にチェックを入れ忘れ → Clone したフォルダが空になる

### `02-clone.md` Cloneする

- VSCode コマンドパレット（`Ctrl/Cmd + Shift + P`）→ `Git: Clone`
- リポジトリ一覧から選択 or URL を貼る（**HTTPS** を推奨、SSHは使わない）
- 保存先フォルダを選ぶ
  - **日本語・スペースを含まないパス**を推奨
- Open でVSCodeが切り替わる

**OS切替が必要な箇所**
- パスの例（`C:\Users\...` vs `/Users/.../`）
- フォルダ選択ダイアログのUI

**よくあるつまずき**
- Clone先をDesktopにしてしまい後で迷子になる → プロジェクト用フォルダを決める

### `03-markdown-edit.md` READMEを書く

- エクスプローラから `README.md` を開く
- 既存の内容を削除
- **テンプレをコピー**（`<CopyBlock>` 配置）

```md
# 👋 こんにちは、<あなたの名前> です

## 自己紹介
- 所属: <学科・学年>
- 好きなこと: <例: ゲーム、アニメ、プログラミング>
- 今回の目標: <例: GitHubを使えるようになる>

## これから作りたいもの
<自由に書いてください>
```

- プレースホルダー（`<...>`）を書き換え
- 保存（`Ctrl/Cmd + S`）

**よくあるつまずき**
- 保存を忘れる → 画面のタブに●がついていたら未保存
- Markdownの記法が崩れる → 見出しの `#` の後は半角スペース

### `04-commit-push.md` Commit & Push

**Gitコマンドは使わずVSCodeのUIで完結**させる（挫折防止）。

- 左のサイドバー「ソース管理」アイコン（枝分かれのマーク）
- 変更したファイルが一覧に出る → `+` でステージング
- コミットメッセージを書く（例: `update README`）
- 「コミット」ボタン
- 「変更の同期」ボタン（または「プッシュ」）
- ブラウザで `https://github.com/<ユーザー名>` を開く
- **プロフィール画面にREADMEが表示されていれば成功**

**ここで「全員で画面を見せ合う」タイミングと対応**（仕様書の成果共有に合わせる）

**よくあるつまずき**
- ステージングし忘れて空コミット
- Push し忘れて「ローカルは変わったのにGitHubに反映されない」→ 同期ボタン
- プロフィール画面で見えない → リポジトリ名がユーザー名と一致しているか・Publicか確認

### `05-challenge.md` ミニチャレンジ（装飾・画像）

- Markdownの装飾（太字、リスト、リンク、見出し）
- 画像の挿入
  - リポジトリにファイルを追加して相対パスで参照
  - または既存の画像URLを貼る
- 絵文字の使い方（`:wave:` 記法）
- 「自分の色」を出す

**よくあるつまずき**
- 画像のパスが間違っている → 相対パスの書き方を例示
- コミット/Push し忘れで反映されない

- [ ] ゴールが**一文**で書いてある
- [ ] 所要時間が書いてある
- [ ] 前のステップへのリンクがある
- [ ] 手順が番号付きリストで、1ステップ1動作
- [ ] OS固有の箇所に `<OSTabs>` が使われている
- [ ] つまずきセクションが最低2項目ある
- [ ] 末尾に `<ProgressCheck>` が置かれている
- [ ] 末尾に次のステップへのリンクがある
- [ ] スクショが入る箇所にalt textが書かれている
- [ ] 外部リンクは `target="_blank"` 相当の案内（VitePress 既定）

## 用語と表記の統一

- 「リポジトリ」「Repository」両方を混在させない → **リポジトリ**で統一
- 「プッシュ」「Push」→ ボタン名は英語のまま、説明文は**プッシュ**
- 「Commit」「コミット」→ **コミット**
- キー表記: `Ctrl` / `Cmd` は両方明記、ショートカットは `Ctrl + Shift + P` 形式
- コードは必ずコードブロック（バッククォート）に入れる
- 画面要素（ボタン名など）は **太字** で表記
